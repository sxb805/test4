# 项目级通用规范（全局生效）

本文件只放跨前后端的通用约定；语言/框架的实现细则请看各子目录 AGENTS 文档。

## 1. 规范分流
1. 后端规范：`backend/AGENTS.md`
2. 前端规范：`frontend/AGENTS.md`

## 2. 跨端协作约定
1. 前后端接口字段命名保持一致，避免同义不同名。
2. 所有业务失败场景必须返回可读 `msg`，前端必须透出。
3. 任何接口变更（路径、入参、出参）必须同步更新前端调用与文档。
4. 导入导出能力默认作为 CRUD 标配能力评估，不可临时遗漏。
5. 新增/修改“列表查询条件”时，必须同步完成查询链路四点对齐：前端查询表单、前端请求参数、后端 `QueryDTO` 字段、后端 `buildQuery` 条件；禁止只改前端不改后端。

## 3. 交付与验收约定
1. 每次 CRUD 改造至少覆盖：列表、查看、新增、编辑、删除。
2. 涉及导入导出时，必须验证成功与失败路径。
3. 提交前必须做最小冒烟，避免“接口通但页面崩”。
4. 严格交付报告必须是“执行证据报告”，不是纯实现说明。
5. 执行证据报告必须落地到 `docs/reports/`，并包含至少以下字段：
   - 实际执行命令（逐条）
   - 实际执行结果（通过/失败）
   - 关键输出摘要（如 `BUILD SUCCESS`、`Tests run`、冒烟脚本通过标识）
   - 失败归因分层（业务问题/环境问题/存量问题）
   - 本次测试时间与环境口径（端口、关键依赖能力）
6. 若同一模块存在“交付说明”与“测试证据”，必须在同一报告中分区呈现；禁止仅提交说明而缺失执行证据。
7. 严格交付默认要求同时记录后端接口冒烟与前端关键路径冒烟的可追溯证据（命令、结果、摘要）。

### 3.1 严格交付报告推荐结构
1. 结论总览（功能可用性/本次改动质量/全局门禁）
2. 执行记录（命令 + 结果 + 关键摘要）
3. 后端接口冒烟记录
4. 前端关键路径冒烟记录
5. 变更清单与契约清单（接口、路由、导入映射、联调说明）
6. 风险与遗留项（如存量 lint、环境依赖）

## 4. 命令与环境约定
1. 在仓库内执行命令时优先使用对应子项目目录。
2. 后端 Maven 命令按 `backend/AGENTS.md` 约定执行。
3. 冒烟执行前必须做服务状态预检（端口/进程），优先复用已运行服务，禁止无检查重复启动同类服务。
4. 若本次改动涉及运行时代码并需要联调验证，必须确认服务已加载最新代码（重启或等效生效确认），并在报告中记录“是否重启、如何确认生效”。
5. 需要鉴权联调时，默认使用前端服务登录接口 `POST /cas/login` 获取 token，请求体为 JSON：`{"username":"xxx","password":"xxx"}`，其中 `password` 必须先做 MD5 后再提交。
6. 每次任务执行前必须预判是否需要 token；若需要但上下文缺失且用户未提供账号/密码，必须先向用户询问后再执行，禁止执行到中途因鉴权缺失中断。
7. 若目标端口已被其他旧进程占用且无法确认已加载最新代码，必须先停止占用进程再重启服务；禁止在端口冲突状态下继续联调并给出“通过”结论。

## 5. 人员树通用约定
1. 项目内“人员树”默认统一使用接口：`/cloud/management/rest/tree/loadStaffTree`。
2. 人员树请求鉴权统一走请求头 `Authorization`（由项目请求层自动注入），业务代码中禁止再拼接 URL 参数 `token`。
3. 前端树选择场景默认仅允许选择 `type=Staff` 节点，部门/组织节点应为不可选。

## 6. 冒烟脚本入口约定（跨窗口默认）
1. 模块全链路冒烟默认入口：`scripts/module-pipeline.sh --module <moduleName>`（包含：自检、后端重启生效、鉴权、后端接口冒烟、前端 E2E、报告生成）。
2. 模块后端接口冒烟入口：`scripts/module-api-smoke.sh --module <moduleName>`（默认串行校验 `save/page/get/delete` 且断言 `result=0`）。
3. 新模块接入时，优先新增模块配置文件 `scripts/modules/<moduleName>.sh`，禁止复制改造整套脚本主体。
4. 兼容历史入口：`scripts/run-taskworkitem-pipeline.sh` 与 `scripts/taskworkitem-api-smoke.sh` 仅作为兼容封装，后续统一收敛到第1/2条通用入口。
