# 前端项目规范（CRUD 生成专用）

本文件仅约束 `frontend` 代码生成与改造；后端规范以 `backend/AGENTS.md` 为准。

## 0. 快速决策（先看这个）
1. 轻表单（字段少、依赖弱）：单列，不强制分组。
2. 常规表单：双列为主，长文本/上传整行；按业务语义分组。
3. 字段密集表单：可三列，但需保证可读性，窄屏自动降级。
4. 有前置依赖流程：用步骤式（`Steps`），分步校验，最终全量校验。
5. 1对多明细录入：用动态 `VtxDatagrid`，行内增删改和行级校验。
6. 状态展示：统一状态标签；有汇总诉求时加统计区并支持状态联动筛选。
7. 导入：默认走“两段式导入”（参数弹窗 + `VtxImport2` 上传）。

## 0.1 UI 对齐优先级（强约束）
1. 必须：当已有同业务页面可参考时，UI 一致性优先级高于“实现更简洁”；禁止擅自替换为通用简版弹窗。
2. 必须：涉及导入、步骤、状态统计、动态明细等高频模块时，优先复用项目内现成实现或同等视觉结构，不得只保留功能。
3. 必须：若需求方给出目标截图，交付以“结构一致、交互一致、文案一致”为验收标准，而非“功能可用即可”。

## 1. 接口与路径硬约束
1. 后端接口统一使用前缀：`/cloud/sample`。
2. 页面 service 必须声明 `API_PREFIX`，禁止在页面中散落硬编码接口。
3. CRUD 基础接口按资源统一：
   - `page`: 优先 `GET ${API_PREFIX}/{resource}/page`（后端为 `Pageable` 时）
   - `get`: `GET ${API_PREFIX}/{resource}/get`
   - `save`: `POST ${API_PREFIX}/{resource}/save`
   - `update`: `POST ${API_PREFIX}/{resource}/update`
   - `delete`: `POST ${API_PREFIX}/{resource}/delete`
4. 导入导出接口默认：
   - `import`: `POST ${API_PREFIX}/{resource}/importExcel`
   - `export`: `POST ${API_PREFIX}/{resource}/exportExcel`
5. 若后端分页使用 Spring `Pageable`，前端参数必须传：`page(0基)`、`size`；禁止使用 `rows`。

## 2. 页面生成标准结构
1. 每个 CRUD 页面固定目录：
   - `src/pages/{resource}/index.js`
   - `src/pages/{resource}/model.js`
   - `src/pages/{resource}/service.js`
   - `src/pages/{resource}/components/Add.js`
   - `src/pages/{resource}/components/View.js`
   - `src/pages/{resource}/components/Export.js`
2. 必须同步注册路由 `config/routes.js`，禁止只建页面不挂路由。
3. `namespace` 必须与页面资源名一致，禁止沿用 demo 的 `user` 等历史命名。

## 3. 日期与表单稳定性约束
1. `DatePicker` 字段回填必须先转 `dayjs`；禁止直接把接口字符串灌入 `DatePicker`。
2. `Form` 禁止使用包含日期字符串的 `initialValues` 直接初始化。
3. 提交保存前必须把日期字段转换为后端约定字符串：
   - 日期：`YYYY-MM-DD`
   - 时间：`YYYY-MM-DD HH:mm:ss`

## 4. 错误提示与交互约束
1. 保存/更新接口返回失败时，必须优先弹后端 `msg`：`message.error(res.msg || "操作失败")`。
2. 删除、导入、导出失败都必须给用户可见提示，禁止静默失败。
3. 页面禁止“只判断布尔成功”；需要兼容后端 `{ result, msg, data }` 返回结构。
4. 若后端导入与表单口径不一致（如：表单传 `managerStaffId`、导入传 `managerStaffName`），页面必须明确提示，避免用户混填。
5. 导入失败时若后端返回错误文件ID（如 `data`），前端必须展示该ID，便于定位错误文件。

## 5. 组件选型约束
1. 优先使用项目中已稳定使用的组件组合（`antd + @vtx/components` 已有用法）。
2. 新组件接入前，先在仓库中检索是否已有同类使用示例；无示例时优先保守实现。

## 6. 新增/修改表单分组规范
1. 必须：新增与修改共用同一套表单组件（同一弹窗骨架），仅通过 `id` 或显式 `mode` 区分提交方法与标题文案，禁止维护两套结构相近的重复表单。
2. 必须：轻量表单（字段少、依赖关系弱）默认使用单列布局（`cols=1` 或等效实现），且可不再额外分组。
3. 必须：复杂表单必须按业务域分组展示，至少采用“一级分组（Card）+ 二级分组（Pane/子标题）”结构，避免所有字段平铺在单层表单内。
4. 必须：分组命名必须直接反映业务语义（如“基本信息”“检测项目”），禁止使用“分组1/分组2”等无语义命名。
5. 必须：布局保持“短字段双列、长字段整行”，备注、附件上传、明细说明等字段默认整行展示，减少换行抖动与阅读跳跃。
6. 必须：动态明细（如检测项目、子项列表）独立成分组，不与基础字段混排；明细项至少具备新增、编辑、删除和行级校验能力。
7. 必须：提交前校验顺序为“基础信息 -> 动态明细 -> 整体提交”，任一分组失败都必须即时提示可读 `msg`。
8. 必须：字段密集场景允许三列布局（单组有效字段数较多且多为短字段时），但必须保证可读性与一致间距；移动端或窄宽度下必须自动降为两列或单列。
9. 推荐：同一分组内优先保持统一列数；仅当文本域、上传区等长字段出现时再跨列整行，避免视觉节奏混乱。
10. 推荐：分组标题支持折叠展开（默认展开核心分组），在字段较多场景下降低首屏压迫感。
11. 推荐：为关键字段补充 tooltip 或示例文案，优先解释命名规则、取值口径、自动带出逻辑。
12. 推荐：动态明细的新增前校验上一条是否已完整填写，避免产生多条“半成品”空行数据。

## 6.1 新增/修改/查看分组统一规范（参考 `samplePointManage`）
1. 必须：`新增`、`修改`、`查看`三种弹窗使用同一业务分组骨架，分组标题、字段顺序、字段归属保持一致。
2. 必须：推荐采用“一级 Card + 二级 Pane”结构：
   - 一级：业务域（如“采样信息”“检测项目”）
   - 二级：子域（如“基本信息”）
3. 必须：`查看`页与`新增/修改`页字段分组保持 1:1 对齐，不得出现“编辑有分组、查看无分组”。
4. 必须：建议统一分组顺序：
   - 第一组：基础主数据（编码、名称、归属、时间等）
   - 第二组：补充信息（备注、说明、附件）
   - 第三组：业务明细（动态 DataGrid、子项配置）
5. 必须：长文本与附件字段归入“补充信息”组并整行展示；明细字段归入“业务明细”组，不与基础字段混排。
6. 推荐：分组标题文案保持稳定，避免同类页面出现“基础信息/基本信息/主信息”混用。
7. 推荐：`查看`模式使用 `VtxFormLayout mode="view"`，并复用同一分组结构只切换渲染模式。

## 7. 步骤式新增/修改规范（按顺序添加）
1. 必须：当表单存在明显前置依赖（如“先基础信息，再生成/填写明细”）时，必须采用步骤式交互（`Steps + 下一步/上一步`），禁止把所有字段一次性平铺。
2. 必须：步骤划分至少包含“基础信息”与“业务明细/记录”两段；每一步标题必须是业务语义名称。
3. 必须：`下一步` 前只校验当前步骤字段，未通过时阻断跳转并提示可读 `msg`；不得跨步骤一次性报错。
4. 必须：最终提交前必须做全量校验与数据组装，避免只校验最后一步。
5. 必须：编辑态（修改）进入步骤弹窗时，必须正确回填并支持直接定位到可编辑步骤；禁止出现“只能从第一步重填”。
6. 推荐：步骤内容容器保持固定区域高度或最小高度，减少切换步骤时弹窗高度剧烈抖动。
7. 推荐：步骤按钮文案统一为“下一步 / 上一步 / 确定 / 取消”，避免同类页面出现多套叫法。

## 8. 复用组件与一致性约束
1. 必须：新增页面优先复用项目现有基础组合：`VtxModal`、`VtxFormLayout`、`VtxSearch`、`VtxDatagrid`、`useForm`，禁止同类页面混用多套风格差异明显的表单/表格框架。
2. 必须：业务选择类字段优先复用现有业务组件，禁止重复造轮子或重写同类下拉逻辑。优先清单：
   - `@/components/business/waterWorks`
   - `@/components/business/pointSelect`
   - `@/components/business/inspectionSelect`
   - `@/components/business/sampleSelect`
   - `@/components/business/indicatorSelect`
   - `@/components/enum-select`
3. 必须：与既有模块同业务含义的字段，组件选型必须一致（如水厂、点位、化验指标），避免同字段在不同页面出现不同交互。
4. 必须：若确需新增组件或改造既有组件，先在页面内给出不可复用原因（字段口径/交互差异/性能限制），并确保不破坏现有页面行为。
5. 推荐：抽离可复用片段时优先沉淀到 `src/components/business` 或公共 hooks，避免只在单页面私有复用。

## 9. 组合字段规范（同组双控件/多控件）
1. 必须：当两个控件共同表达一个业务字段时（如“循环周期=数值+单位”），应封装为一个组合组件并挂在同一个 `FormItem` 下，禁止拆成两个无关联字段。
2. 必须：组合字段值使用对象结构统一承载（如 `{ cyclePeriod, cyclePeriodType }`），提交前在页面层映射为后端所需平铺字段。
3. 必须：组合字段的校验在同一 `FormItem` 内完成，需覆盖“子项缺失/部分填写”场景，避免只校验其中一个控件。
4. 推荐：组合控件内部使用统一布局（如 `flex` 等分）保证视觉连续性，减少用户把它误解为两个独立字段。
5. 推荐：同类业务字段在项目内保持同一交互结构与命名（如周期值/周期单位），避免跨页面认知切换。

## 10. 常规表单元素规范（高频）
1. 必须：基础信息表单默认覆盖文本输入、下拉/下拉树、长文本备注、图片上传四类常见控件，除非业务明确不需要。
2. 必须：树形选择场景优先使用可搜索树选择（如 `TreeSelect`），并保持与现有“水厂/组织”选择交互一致。
3. 必须：备注类字段使用长文本输入（`TextArea`）并整行展示，避免与短字段混排导致可读性下降。
4. 必须：上传图片字段需在标签侧提供明确约束提示（`?` tooltip），至少包含“数量上限 + 单文件大小上限”。
5. 必须：上传组件实际限制与提示文案保持一致，禁止仅展示提示但不做限制。
6. 必须：普通文本字段使用 `Input`，数字字段使用 `InputNumber`（或等效数字输入组件），禁止用普通文本输入承载数值字段。
7. 必须：枚举值字段优先使用下拉选择（`Select/EnumSelect`）；二值互斥选项优先使用单选（`Radio`）明确表达。
8. 必须：数字字段需配置最小值/精度/格式校验（按业务需要），并在提交前保证类型正确。
9. 推荐：输入/选择框占位文案统一使用“请输入/请选择+字段名”。
10. 推荐：字段说明（`?` tooltip）优先放在存在口径歧义或限制条件的字段上，避免信息噪音。
11. 必须：图片上传与回显优先复用项目既有模式（`VtxUpload` + `picture-card`），新增/编辑与查看保持同一字段（如 `pictureFile`）口径。
12. 必须：查看态图片回显使用项目当前稳定写法（如 `VtxUpload viewMode`）；禁止在不同页面混用不一致的查看属性。
13. 必须：上传返回对象序列化前保持与参考实现一致的结构，避免擅自裁剪导致回显异常。
14. 必须：若出现“文件名可见但缩略图破图”，优先排查网关/代理/鉴权链路（文件预览地址可访问性），再排查页面组件写法。
15. 推荐：图片上传问题排查顺序固定为：`上传返回结构 -> 保存字段内容 -> 回显解析 -> 代理/权限 -> 组件属性`。

## 11. 动态增减 DataGrid 明细规范
1. 必须：当业务存在“1对多明细录入”（如检测项目列表）时，优先使用 `VtxDatagrid` 承载行内编辑，不用散落的重复表单块。
2. 必须：每行必须有稳定 `rowKey`（建议前端生成 uuid），避免新增/删除后行状态错乱。
3. 必须：新增前校验当前编辑行完整性；存在未保存行时禁止继续新增并提示可读 `msg`。
4. 必须：删除最后一行时要有保护（如至少保留1行或给出业务提示），避免出现空明细非法提交。
5. 必须：行内字段支持依赖联动（如选择指标自动带出单位），并在同一行内即时刷新。
6. 必须：提交前先完成明细级校验（必填、区间、逻辑比较），再进入整体保存。
7. 推荐：明细区底部统一提供“+ 添加xxx”按钮，样式与文案跨页面保持一致。
8. 推荐：复杂明细字段用组合控件（如“符号+数值+单位周期”）降低列数，避免横向滚动过重。

## 12. 导入标准流程（默认采用）
1. 必须：业务文案统一使用“导入”；默认仅实现“标准导入”路径。
2. 必须：禁止在导入弹窗中出现“标准导入/简版导入”单选、切换或额外入口，除非需求明确要求保留双模式。
3. 必须：默认优先采用 `enterpriseList` 形态的“单入口导入”：点击“导入”直接打开唯一 `VtxImport2` 弹窗，不增加中间选择弹窗。
4. 必须：仅当导入前置参数确实不可省略时，才允许增加参数弹窗（两段式）；否则一律直开 `VtxImport2`。
5. 必须：模板地址支持固定模板或后端下载模板接口；若仅一种业务模板，禁止再做模式分支。
6. 必须：上传请求统一携带基础参数 `tenantId`、`userId`，并按场景附加业务参数。
7. 必须：上传结果处理以后端 `result` 为准；`result === 0` 时刷新列表与统计，关闭导入弹窗。
8. 必须：失败路径必须透出后端错误信息，并支持错误文件下载（通过统一 `errorURL`）。
9. 必须：若页面已有“标准导入样式”（步骤卡片式：下载模板区 + 选择文件区 + 底部按钮），则该页面后续改造必须保持该样式，不得回退为“选择文件 + 下载模板”简版弹窗。
10. 必须：导入弹窗标题遵循“{业务名称} > 导入”结构；按钮顺序统一“关闭 / 上传”。
11. 必须：导入区文案需明确“支持excel格式”等格式说明；下载入口与上传入口分区展示，禁止混在同一行弱化指引。
12. 必须：使用 `VtxImport2` 时，`title` 仅传“业务名称”（如“样例”）；组件会自动渲染为“{title} > 导入”，禁止手动传“xx > 导入”导致标题重复。
13. 必须：`VtxImport2.afterUpload` 回调需兼容 `object|string` 两种返回；字符串解析必须使用 `try/catch` 兜底，禁止直接 `JSON.parse` 导致页面崩溃。
14. 必须：`errorURL` 使用“项目前缀 + `/common/downloadImportExcel`”完整后端路径（如 `/cloud/sample/common/downloadImportExcel`），禁止使用裸相对路径（如 `common/downloadImportExcel`）。
15. 必须：若已使用 `VtxImport2` 内置失败结果页与错误文件下载入口，页面层不得再重复弹“错误文件ID”提示，避免重复/冲突反馈。
16. 必须：实现前先对照 `web-frontend/src/pages/enterpriseList/index.tsx` 的导入实现；无特殊需求时按其交互与结构 1:1 复用。
17. 禁止：出现“导入类型”字段、`Radio` 导入模式选择、或“标准导入/简版导入”任一文案。
18. 禁止：新增第二个导入按钮、二级导入入口、或同页并存两套导入组件。

## 12.1 导入实现硬验收（提交前自检）
1. 必过：页面上仅有一个“导入”入口按钮。
2. 必过：点击“导入”后直接打开唯一导入弹窗（`VtxImport2`），无中间模式选择弹窗。
3. 必过：代码中不存在“简版导入”“导入类型”“standard/simple”等模式分流字段。
4. 必过：导入成功分支包含 `result===0` 后刷新列表（及统计）并关闭弹窗。
5. 必过：导入失败分支可见后端 `msg` 且可下载错误文件（`errorURL`）。

## 13. 列表统计区规范（可选能力）
1. 可选：当列表存在“总量 + 分类数量”类业务诉求时，页面可在查询区与表格区之间增加统计区（如总数、分类数、状态数）。
2. 必须：统计区数据口径与当前筛选条件一致；用户点击“查询/重置”后统计区必须同步刷新，禁止出现“表格已变、统计未变”。
3. 必须：统计项命名与后端字段语义一致，避免同义不同名；数量展示统一为数值+单位（如“个”）。
4. 必须：当统计区提供状态项时，状态项可点击并联动过滤下方列表（再次点击可取消或切换），并提供当前选中态样式。
5. 必须：统计区遵循“总数 + 状态分组”层级，二者之间保留明显分隔（如竖线）以表达分组关系。
6. 必须：状态项与表格状态标签使用同一颜色语义映射（例如：已完成=绿色、待处理=黄色、异常=红色），禁止同状态多色。
7. 推荐：总数置于分组首位，状态项按业务流程顺序排列（如未检测 -> 已检测）。
8. 推荐：统计项被选中时可增加下划线/高亮边框等弱强调，避免与按钮态混淆。
9. 推荐：统计区采用横向卡片/分栏样式，按“总数 -> 重点分类”顺序排列，首项优先展示总量。
10. 推荐：统计项数量建议控制在 4~6 个，过多时按业务优先级折叠或分组，避免挤压列表主体区域。
11. 推荐：统计区作为可开关能力接入（配置开关或按页面需求启用），不强制所有 CRUD 页面默认展示。

## 14. 列表状态标签规范
1. 必须：列表中所有“状态类字段”（如采样状态、检测状态、审核状态）统一使用标签样式展示，禁止纯文本裸露状态值。
2. 必须：同一页面多个状态列可并列展示标签，但颜色体系必须统一且可被快速区分。
3. 必须：状态文案使用后端返回的业务语义值（`xxxValue/msg`），避免前端二次造词导致歧义。
4. 推荐：状态标签使用项目已有组件（如 `@/components/status-tag`）或同等样式能力，减少页面实现差异。

## 15. 标签页与查询区联动规范
1. 必须：当标签页位于查询区上方时，默认各标签页不复用同一查询条件；切换标签页应切换到各自独立的查询状态。
2. 必须：当标签页位于查询区下方时，默认各标签页复用同一套查询条件；切换标签页后保留当前查询输入并沿用筛选结果。
3. 必须：页面开发前先明确“标签页在查询区上/下”的布局，再确定查询状态作用域（独立或共享），禁止实现与布局语义相反。
4. 推荐：在代码中显式区分 `sharedSearchState` 与 `tabSearchStateMap` 两种模式，避免后续维护误改。
5. 推荐：切换标签页时保持统计区、列表、分页与当前作用域的查询状态一致刷新。

## 16. CRUD 冒烟验收清单（提交前必过）
1. 列表可加载，分页正常。
2. 新增：成功提示正确；失败可见后端 `msg`。
3. 编辑：打开不崩溃，保存后页面不白屏，列表可刷新。
4. 查看：弹窗可打开，字段显示正常。
5. 删除：可成功删除并刷新。
6. 导入：成功与失败路径都能提示，失败可下载错误文件（若后端提供）。
7. 导出：全部/选中/当前页均可下载。
8. 分页联调校验：避免出现 `total>0` 但 `rows=[]` 的参数错配情况。
9. UI 对齐校验：若需求附参考图，必须逐项核对弹窗结构、标题、按钮顺序、关键文案与交互触发点。

## 17. 生成提示词最小必填信息（给 Agent/Skill）
1. 资源名（如 `example`）、中文标题（如“样例”）。
2. 后端 context-path（本项目固定 `/cloud/sample`）。
3. DTO/VO/Query 字段清单（含日期字段类型）。
4. 导入导出接口与模板地址。
5. 成功/失败消息策略（失败透出后端 `msg`）。
